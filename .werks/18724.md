This werk introduces monitoring support for Podman containers and pods. The new monitoring includes both host-level and per-container services.

### Host-level Services
The following services are created on the Podman host:<br>
- **Podman containers**: Monitors the total number of containers in different states (running, created, paused, stopped, restarting, removing, dead, exited, exited with non-zero exit code) with configurable upper and lower thresholds per state<br>
- **Podman pods**: Monitors the total number of pods in different states (running, created, stopped, dead, exited, paused, degraded) with configurable upper and lower thresholds per state<br>
- **Podman disk usage**: Monitors disk space usage for containers, images, and volumes. Reports total size, reclaimable size (containers and volumes), total count, and active count per entity type<br>
- **Podman status**: Reports any errors encountered when querying the Podman API or CLI

### Per-container Services (Piggyback)
Each container is monitored as a separate piggyback host with the following services:<br>
- **Status**: Monitors the operational state of the container with configurable state-to-monitoring-state mappings for each container state (created, running, paused, restarting, removing, exited with zero, exited with non-zero, dead). Shows pod membership if applicable<br>
- **Health**: Monitors container health check status (healthy, starting, unhealthy, no healthcheck configured) with configurable state mappings. Details include last health report output, health check command, consecutive failing streak, on-failure action, and last exit code<br>
- **Restarts**: Tracks total container restart count and restarts in the last hour with configurable upper thresholds for both<br>
- **CPU utilization**: Monitors container CPU usage percentage with configurable thresholds<br>
- **Memory**: Monitors container memory usage with percentage and absolute thresholds<br>
- **Container IO**: Monitors disk I/O operations (read/write operations per second)<br>
- **Uptime**: Tracks how long the container has been running

### Inventory Information
The monitoring also provides inventory data for:<br>
- **Podman engine**: Mode (rootless/rootful), version, registries, and running/paused/stopped/exited container counts and image count<br>
- **Podman containers**: ID, creation time, name, labels, status, and image per container<br>
- **Podman images**: ID, creation time, size, container count, repository, and tag per image<br>
- **Per-container details**: Hostname, pod, labels, and network configuration (IP address, gateway, MAC address)

### Configuration
Podman monitoring is configured via the "Podman hosts and containers (Linux)" agent rule.<br>
The monitoring supports two connection methods:<br>
- **API mode** (default): Connects to Podman via Unix sockets. Supports automatic socket detection for root sockets, user sockets, or both, as well as manual socket path specification<br>
- **CLI mode**: Invokes the `podman` binary directly. Automatically discovers users running Podman containers and queries each user's containers. Useful when sockets are not available

Piggyback host naming is configurable with three methods:<br>
- `nodename_name` (default): `{podman_hostname}_{container_name}`<br>
- `name`: Container name only<br>
- `name_id`: `{container_name}_{short_container_id}`

### Prerequisites
- Python 3.8 or newer must be available on the monitored host<br>
- The `requests` Python library is required for API mode (e.g. `python3-requests` OS package or `pip install requests`). If unavailable, use CLI mode instead<br>
- Only cgroup v2 is supported for monitoring
