# omd: Local privilege escalation when executing omd commands as root

---------- | ---

version    | 2.6.0b1
class      | security
edition    | community

compatible | no

A critical privilege escalation vulnerability existed when executing the administrative `omd` commands with root privileges.
A [site user](https://docs.checkmk.com/latest/en/omd_basics.html#site_user) could exploit this mechanism to escalate their privileges to root.
This escalation could also be triggered automatically by standard system operations such as post update triggers.

Calling `omd` inside the site context is safe, as there is no path for the site user to escalate to root from within it.

*Update instructions*:

Do **not** use `omd` as a root user to interact with your site until you have installed a patched version of `Checkmk`.
(Note: Running the standard package installation first, followed by the site update procedure, is safe and will not trigger the vulnerability).

Installing the update automatically sets the patched `omd` as your default system command, which mitigates the primary attack vector.

While the new `omd` executable prevents the invocation of vulnerable code, there is no way for us to verify that the old executables won't be invoked directly or triggered by other mechanisms on your system.
Therefore, we strongly recommend updating all sites and completely removing the vulnerable packages from your system.

*Incompatible - Manual interaction required*:

To protect your system, `omd` will now explicitly refuse to execute commands that would trigger a switch to an older, vulnerable version.
Because of this protection, global commands (such as `omd mv` or `omd restore`) will simply be refused and no longer work against sites running older versions.
Update affected sites to a secure version to use these commands again.

*Question*: What is the vulnerability and who is potentially affected?

*Answer*: This is a local privilege escalation.
A user who already has access to modify the site context can place a malicious payload.
When the `omd` command is subsequently executed by root, that payload is executed with root privileges, allowing the site user to become root.

The site user serves as an isolation layer between the host machine and the Checkmk instance.
The following are some practical examples:

* **Shared Hosts**: If multiple sites share a server, a compromised site user can escalate to root, breaking isolation and taking over the entire host and all other sites.
* **Restricted Hosts**: If you deny root access to site administrators but allow tasks like uploading MKPs, they can exploit this to bypass your restrictions and gain full root access.
* **Dedicated Hosts**: If a single site runs alone on a server, the severity is lower. This is strictly a local privilege escalation for an already-compromised site user, not an external entry point.

*Question*: Is it still possible to interact with your site by performing actions as a site user?

*Answer*: Yes, the `omd` is always safe to run as a site user.
For example, this command is safe

```sh
su - <site name> -c "omd stop"
```

*Question*: How can I verify whether the `omd`, which is currently in my path, is vulnerable?

*Answer*: This can be verified by checking the new flag file, i.e.:

```sh
VERSION=$(omd version -b)
if [ -f "/omd/versions/$VERSION/share/omd/security-werk-18891.flag" ]; then
  echo "System is PATCHED."
else
  echo "System is VULNERABLE."
fi
```

*Question*: How can I check which of my installed versions are affected?

*Answer*:

```sh
for path in /omd/versions/*; do
  VERSION=$(basename "$path")
  if [ -f "$path/share/omd/security-werk-18891.flag" ]; then
    echo "$VERSION is PATCHED"
  else
    echo "$VERSION is VULNERABLE"
  fi
done
```

*Question*: Is it sufficient to install the new version of Checkmk?

*Answer*: No, but it is a critical first step that significantly reduces your immediate risk.
Installing the patched package replaces the default system `omd` command.
For most environments, this protects the system against standard exploitation and gives you time to plan your site updates without operational stress.
However, as long as older packages remain installed, the vulnerable executables still exist on your system.
For example, if your environment relies on custom scripts or cron jobs, which use an absolute path to an older `omd` version, the vulnerability can still be exploited.
Furthermore, installing older Checkmk packages in the future will inadvertently revert the default `omd` command back to a vulnerable state.
In these cases, your system is only protected, if all of your sites are up-to-date.

*Affected Versions*:

* 2.4.0
* 2.3.0
* 2.2.0 (EOL, please upgrade to 2.3.0 or newer)
