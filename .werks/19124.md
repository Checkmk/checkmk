# Fix stuck averaging values after time source change/reset

key        | value
---------- | ---
date       | 2026-02-16T13:55:09+00:00
version    | 2.4.0p22
class      | fix
edition    | cre
component  | checks
level      | 1
compatible | yes


When the time source of a monitored device changed (e.g. from epoch
timestamps to sysUpTime after a 2.2 to higher version migration), the averaging
calculation could get stuck. The backward time jump caused `get_average`
to return the last computed average indefinitely without ever updating
the stored time references, so the average would never incorporate new
values.

The time references in the value store are now reset when a backward
time jump is detected, while preserving the previously computed average
as a seed. This allows averaging to resume normally with the new time
source without losing history.
