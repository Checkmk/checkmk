# ServiceNow notification plugin: filter the issue lookup query

key        | value
---------- | ---
date       | 2026-05-08T13:13:21.474768+00:00
version    | 3.0.0b1
class      | feature
edition    | pro
component  | notifications
level      | 1
compatible | no

When a Checkmk notification fires, the ServiceNow plugin first looks up
whether an issue (incident or case) for the given Checkmk problem ID
already exists. On large ServiceNow instances this lookup can run over
the entire incident or case table and time out, because it is filtered
only by the Checkmk problem ID encoded in the work notes.

A new optional parameter <i>Additional query filters</i> has been added
below the <i>Use site ID prefix</i> field. It accepts a list of
name/value pairs that are AND-combined onto the ServiceNow
<tt>sysparm_query</tt> used during the lookup, e.g.
<tt>company=&lt;sys_id&gt;</tt> or
<tt>assignment_group=&lt;sys_id&gt;</tt>. This narrows the search to the
scope managed by Checkmk and avoids the timeouts.

The same notification macros that are available in <i>Custom fields</i>
can also be used in the filter values.

To keep the lookup symmetric, the configured name/value pairs are also
written to the issue at creation time. Recovery, acknowledgement and
downtime notifications therefore reliably find the previously created
issue without any additional configuration in <i>Custom fields</i>.

<b>Incompatibility — migration of existing open issues:</b> open
ServiceNow issues created before the filter was activated do not carry
the filter fields. After enabling <i>Additional query filters</i> the
lookup will no longer find them, so recovery, acknowledgement and
downtime notifications for those legacy issues will fail with
<i>"No related open issue could be found"</i>.

Before activating the filter, drain or backfill the still-open Checkmk
issues. A ServiceNow Background Script that updates every open issue
whose work notes contain <tt>Check_MK Problem ID:</tt> with the same
field/value pairs is sufficient, e.g.

<pre>
var gr = new GlideRecord('incident');
gr.addQuery('active', true);
gr.addQuery('work_notes', 'CONTAINS', 'Check_MK Problem ID:');
gr.query();
while (gr.next()) {
    gr.company = '&lt;sys_id&gt;';
    gr.update();
}
</pre>

Alternatively, close the affected issues manually. Once all
pre-existing issues are drained, no further action is needed.
